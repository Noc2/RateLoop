# RateLoop Workspace Connection

Complete the connection from the user's one-time message. Treat installation, host trust, and OAuth consent as host-owned security controls, but continue automatically after any required native action succeeds.

## Connection workflow

1. Locate the complete connection URL in the user's message. Accept only HTTPS URLs whose origin is exactly `https://rateloop-tokenless.vercel.app`, whose path starts with `/connect/aci_`, and whose fragment is non-empty.
2. Parse and validate the URL locally. Never open, fetch, log, quote, or reproduce the complete URL. The fragment is single-use claim material and must not enter an HTTP request, shell argument, repository, ordinary response, or diagnostic output.
3. Check the current tool inventory for the exact `rateloop-workspace` tools before making any connection claim. The separate public `rateloop` server cannot connect a private workspace. A plugin row, an enabled toggle, a completed browser redirect, or the absence of an error is not evidence that the workspace tools are available.
4. The connection message deliberately combines a structured **RateLoop Workspace** plugin mention with an explicit `$rateloop-workspace-connection` invocation. The dedicated plugin contains only the OAuth-protected `rateloop-workspace` server, so the mention installs or activates the correct server and its native install/connect flow whether this is a first connection or a replacement for a deleted workspace. If the host actually presents an install, trust, update, OAuth, **Continue**, restart, or new-task action, state or follow only that exact action while preserving the original intent. Never invent a button, menu, settings path, or host capability. Never rewrite MCP configuration by hand or substitute the public server.
5. Treat the first missing workspace-tool inventory for a connection intent as activation pending, including when the active turn resumed after host setup. Preserve the intent and never diagnose a stale install or recommend uninstall from that first check. Follow only a native action the host actually presents. If none is visible, say only: "The RateLoop connection is pending host activation. I’ll re-check the protected workspace tools on the next active turn." Do not ask the user to start a new task, paste or resend the connection message, or uninstall anything.
6. After the host reports setup complete or offers **Continue**, resume through that native action. On the next active turn after the first missing-tool check, check the workspace tool inventory once more. Do not infer OAuth success or tool availability from the user's return. Do not run a second login, launch `codex exec` or `codex resume`, start a nested agent, replay the conversation, or use shell and self-message workarounds.
7. As soon as the workspace tools are available, prefer `rateloop_connect_workspace` with `{ "connectionUrl": "<complete URL>", "reportedLane": "plugin-with-hooks" }`. `reportedLane` is this plugin's honest self-declaration that its skills and hooks run on this host; RateLoop records it as host-reported, never as verified. This protected one-call workflow is the only permitted transfer of the complete URL. It idempotently resumes any partial claim, loads the canonical policy context, and verifies the connection without returning the URL. Treat `connected: true` with a successful `verification` as complete, including an idempotent retry, and follow its machine-readable `nextAction`.
8. If `rateloop_connect_workspace` is unavailable, use the granular fallback: call `rateloop_claim_connection_intent` with the same protected arguments (including `"reportedLane": "plugin-with-hooks"`), call `rateloop_get_agent_context` and adopt the returned workspace, workflow, publishing, and human-review policy, then call `rateloop_verify_connection`. Treat an idempotent already-claimed response as resumable success. These fallback calls are recovery primitives, not the preferred path.
9. If the one-call workflow reports a retryable interruption after a partial step, retry `rateloop_connect_workspace` with the same privately held URL; its claim, context load, and verification are idempotent. Never print or reconstruct the URL while retrying. Verification must not create a review opportunity or change adaptive-review evidence.
10. When the preferred workflow or fallback verification succeeds, report only that the workspace is connected and ready. Do not repeat identifiers or connection material unless the tool explicitly returns a display-safe value intended for the user.

## Recovery boundaries

- Resume automatically after a host-presented setup or continuation action. Do not ask the user to paste the connection message again while the intent remains valid.
- A deleted RateLoop workspace revokes its OAuth token family. The next RateLoop Workspace invocation should make the host observe the protected server's authorization challenge and, when required, present one fresh OAuth action. Existing RateLoop plugin installs can retain a stale bundle or authorization binding, so do not assume an enabled plugin row proves the protected tools are available.
- After confirmed OAuth success, never tell the user to approve the same authorization again unless the server returns a new display-safe authorization-required recovery action. If no prompt is visible, do not claim that one is pending.
- Only if a later active turn still lacks the workspace tools after the first missing-tool check, and the host exposes no native action, say exactly: "The RateLoop workspace tools are still unavailable. Uninstall every existing RateLoop plugin, including rateloop and rateloop-workspace, then resume this same task. Do not paste the connection link again." Preserve the original connection intent and re-check the tools on the next active turn. Never tell the user to reinstall a plugin, start a new task, or repeat that uninstall recovery for the same intent. Never tell them to remove unrelated plugins or guess a reload, restart, toggle, gear, or settings path.
- If the user asks whether the connection works now, check the current workspace tool inventory. When the tools exist, prefer the atomic connection tool and use claim -> context -> verification only as its fallback; otherwise report only the unverified tool state and any action the host actually presents.
- Never report OAuth success unless the host returned it. Never report the workspace connected or ready unless `rateloop_connect_workspace` returned `connected: true` with a successful `verification`, or the fallback `rateloop_verify_connection` succeeded.
- Never inspect or replay task previews, terminal history, logs, or diagnostics that may echo the complete connection URL.
- Never create a heartbeat, monitor, background service, scheduled task, or chat polling loop for connection state.
- Never poll registration status or ask for separate workspace-owner approval for the pre-authorized safe profile.
- If the link is invalid, expired, consumed by another installation, or bound to a different workspace, return the tool's display-safe recovery action. Do not improvise a credential or configuration workaround.
- If organization policy blocks MCP installation or OAuth, say so directly. Do not weaken authentication or route workspace work through the public handoff server.

Connection alone does not create a background hook, scheduler, or output interceptor. After connection, use `$rateloop-human-review-loop` to actively evaluate eligible outputs and complete the bound review flow whenever the returned policy requires it.
