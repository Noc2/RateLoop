## New findings

### 6P-1 (Medium, confidence High). Public-rating cursor still permanently wedges after a voting-engine rotation — 5P-2 was not actually fixed

- **Where:** `ContentRegistry.sol:1448-1458` (`advanceRatingSnapshotCursor` passes the *current* `votingEngine`); `ContentRegistryRatingSnapshotLib.sol:298-299` (`votingEngine.roundCore` for rounds with no recorded pending); `RoundLib.sol` (`RoundState` enum zero = `Open`); `ContentRegistry.setVotingEngine` has no per-content terminal-round guard. Re-verified directly by the lead: the no-pending branch reads `roundCore` on the engine handle passed from `ContentRegistry`, which is always the current one.
- **What `03d32a144` fixed vs. didn't:** the commit made *pending* settlements rotation-safe (skip predicate, `sourceReadyAt`, and apply all use the pinned `pending.votingEngine`). But `Cancelled`/`Tied`/`RevealFailed` rounds never record a pending, so the cursor's only way past them is `roundCore` on the current engine — which, after a rotation, returns an empty slot for a round that lived on the previous engine, decoding to `RoundState.Open` (enum zero), never skippable.
- **Scenario:** round 5 ends refund-terminal on engine A before the cursor reaches it; governance rotates to engine B (a supported flow; terminal rounds on A do not trip the `ActiveRoundOnPreviousEngine` guard); round 6 settles on B. `applyRatingPayoutSnapshot(…, 6, …)` reverts `RatingSnapshotOutOfOrder(5)` forever and `advanceRatingSnapshotCursor` sees round 5 as `Open` forever. The content's canonical rating, `lowSince` timer, and slash eligibility freeze permanently, with no governance override (`repointPendingRatingClusterPayoutOracle` requires `pending.exists`). This is the one permanent, no-escape wedge found in the pass, and the seventh-pass audit's review of `03d32a144` did not catch that 5P-2 stayed open — worth correcting in the audit trail.
- **Fix (redeploy is free):** in `_canAdvanceRatingSnapshotCursor`, fall back to `contentRoundTrackingEngine[contentId]` (or a per-round owning-engine record) when the current engine reports `Open` with zero start time; or record a trivially-skippable pending for refund-terminal rounds; or add a CONFIG_ROLE skip that verifies terminal state against a caller-supplied engine with an authorized callback generation.

### 6P-2 (Medium, confidence High). Repeated permissionless reopen re-bumps `claimDeadline` — indefinite recovered-refund griefing

- **Where:** `QuestionRewardPoolEscrowRecoveryLib.sol:239-296` (`reopenRecoveredSnapshotRound`) — re-verified by the lead: there is **no** `!reopenedRecoveredRound[rewardPoolId][roundId]` guard, and the deadline bump `if (block.timestamp > rewardPool.claimDeadline) rewardPool.claimDeadline = block.timestamp` re-executes on every call. Bundle mirror: `QuestionRewardPoolEscrowBundleRecoveryLib.sol:260-314`, reachable repeatedly via `recoverOrReopenSnapshotRoundSet` because `rejectedRecoveredBundleRoundSet` stays true until qualification or abandonment.
- **Scenario:** round N is recovered; a replacement finalizes outside veto and un-rejected but is *unqualifiable* (rawEligibleVoters drift, effective-floor, or "Small allocation" — exactly the gap 6P-4 leaves open). Reopen succeeds and sets `claimDeadline = now`. The funder must then wait `claimDeadline + 7 days` before `refundExpiredRecoveredRewardPool` / `refundRecoveredQuestionBundleReward`. Any address can call reopen again after each bump — all preconditions still hold — resetting the 7-day clock. One cheap transaction per week postpones the refund forever; exit requires governance to reject the replacement root. The eighth-pass audit noted a *single* bump "can delay" refunds; the repeatable bump makes the delay unbounded and attacker-controlled.
- **Fix:** `require(!reopenedRecoveredRound[rewardPoolId][roundId], "Already reopened")` (and the bundle mirror), or bump the deadline only on the first transition to reopened. Closing 6P-4 independently removes the unqualifiable-snapshot fuel.

### 6P-3 (Medium, confidence High). RBTS no-forfeit timeout can be atomically unlocked by rejecting/invalidating a stalled snapshot past `readyAt + 1h`

- **Where:** `RoundVotingEngineRbtsSettlementModule.sol:41-54, 133-157`; `ClusterPayoutOracle.sol:676-702` (`invalidateObjectivelyInvalidRoundPayoutSnapshot`). The blocking half of SC-6P-4 is correct — Proposed, Challenged, and finalized-in-veto snapshots all block the timeout. The gap is the *unblocking* half: the timeout clock is anchored solely to `roundClusterPayoutReadyAt`, and nothing restarts or defers it when a live snapshot leaves the live set via rejection or invalidation.
- **Two concrete paths:**
  1. **Challenged path (deterministic):** an RBTS snapshot is challenged in its 15-minute window; the arbiter is the 2-day timelock, so rejection always lands well past `readyAt + 1h` and the rejection transaction is publicly queued days ahead. Anyone can back-run `rejectRoundPayoutSnapshot` in the same block with `applyRbtsSettlementSnapshot(contentId, roundId, [], [])`, completing settlement no-forfeit before the keeper can broadcast a corrected proposal. Voters who should forfeit escape; honest voters lose the forfeit pool and correlation-weighted rewards.
  2. **Invalidation path:** an eligible frontend squats the slot with an objectively malformed snapshot while the honest keeper is absent; once `readyAt + 1h` passes it bundles `invalidateObjectivelyInvalidRoundPayoutSnapshot` + the empty-input timeout in one transaction.
- **Inconsistency:** this contradicts the plan's own requirement ("the timeout must not create a gas race against a real snapshot that is about to become usable"; "if a snapshot is challenged, keep the round in a disputed state rather than silently timing out"). The disputed state is honored only while status is `Challenged`, not through its resolution. The sibling rating path already has the right shape — its skip requires `proposedAt + 1h` to elapse (`ContentRegistryRatingSnapshotLib.sol:323`); the RBTS module has no equivalent.
- **Fix:** mirror the rating lib — require `block.timestamp ≥ readyAt + TIMEOUT` **and**, if any proposal ever existed for the key, `block.timestamp ≥ proposedAt + TIMEOUT`; ideally record a `rejectedAt` on the oracle and require `≥ rejectedAt + grace` so a rejection at `T` never unlocks the timeout before `T + grace`.

### 6P-4 (Low–Medium, confidence High). L-8P-1 reopen fix (working tree) still misses three qualifier preconditions; the same drift also hard-reverts bundle refunds

- **Where:** working-tree diff to `QuestionRewardPoolEscrowRecoveryLib.sol:272-281` adds consumer-match, `sourceReadyAt != 0`, and `proposedAt ≥ sourceReadyAt && ≥ pinnedAt`, matching part of the qualifier. Still missing, relative to `QuestionRewardPoolEscrowQualificationLib.sol:839-846` and the live recount: (1) `payoutSnapshot.rawEligibleVoters == baseRawEligibleVoters` (drift on a post-finalization ban/credential lapse); (2) the effective floor `effectiveParticipantUnits ≥ floor && totalClaimWeight > 0`; (3) allocation sufficiency. The bundle reopen path shares gaps (1)–(3).
- **Related bundle hard-revert:** `QuestionRewardPoolEscrowBundleActionsLib.sol:1159` `require(payoutSnapshot.rawEligibleVoters == completerCount, "Cluster snapshot mismatch")` sits inside `_qualifyBundleRoundSet`, which `refundQuestionBundleReward`, `claimQuestionBundleReward`, and `syncBundleQuestionTerminal` all call for the head round set. If a completer is banned after the head set's snapshot finalizes, the recount mismatches and *every* refund attempt reverts — including `refundRecoveredQuestionBundleReward` — until the arbiter rejects the root. The abandonment predicates model drift; the inline qualification path reverts instead. This is a liveness-until-governance inconsistency, not funds loss.
- **Fix:** reuse the existing `previewRoundQualificationWithClusterSnapshot` / `hasQualifiableRecoveredBundleReplacementSnapshot` predicate as the reopen precondition on both sides; in `_qualifyBundleRoundSet`, treat a count mismatch as unqualifiable (return pending/false) rather than `require`-reverting, so refunds and the cursor can classify it.

### 6P-5 (Low, confidence High). 1-hour frontend fee withdrawal cannot back payout-root accountability against fresh offenses — contrary to the README/oracle-NatSpec claim

- **Where:** `FrontendRegistry.sol:55-61, 375-414, 543-592` (`FEE_WITHDRAWAL_DELAY = 1 hours`; only slash path is `GOVERNANCE_ROLE` = the 2-day timelock); `README.md:48`; `ClusterPayoutOracle.sol:36` NatSpec ("future fee loss"). The slash sweep itself is correct (confiscates fees + pending withdrawal, blocks completion when slashed). But an operator can propose a bad root, call `requestFeeWithdrawal` in the same block, and complete it 1 hour later — days before any slash for that offense can execute through governance. A rolling hourly withdrawal keeps the slashable fee buffer at ~1–2 hours of accrual permanently; only the 1,000 LREP stake (14-day unbonding) genuinely backs fresh offenses.
- **Inconsistency:** the README's "earned-but-undelivered fees back the operator's payout-root accountability" and the oracle's "future fee loss" overstate what the 1-hour posture can deliver. The 1-hour window is the owner's product decision; the finding is the mismatch between the claim and the achievable slash latency.
- **Fix (pick one):** soften the README/NatSpec to "stake backs accountability; fees are slashable only for offenses adjudicated before withdrawal"; or freeze `completeFeeWithdrawal` while the operator has any live `Challenged` oracle proposal (challenges land within 15 minutes; cheap on-chain check via the registry↔oracle link), which makes the 1-hour fee buffer real.

### 6P-6 (Low, confidence Medium). Seed v4 residuals — Base sequencer grind and a timed stale-seed re-roll ratchet — should be named trust assumptions

- Seed v4 (`RoundRevealLib.sol:281-343`, armed in `settleRound` as `block.number + 1`, resolved via `blockhash` with an EIP-2935 fallback) is a genuine close of SC-5P-1: at reveal time nobody can know the future `blockhash(seedBlock)`, so the pivotal-revealer favorable-branch choice is gone, and the EIP-2935 staleness path is handled with a permissionless re-arm (no v1-style wedge). Two residuals remain:
  1. **Base sequencer grind (accepted-class):** `blockhash(seedBlock)` on an OP-Stack chain is entirely sequencer-constructed; a colluding sequencer can cheaply sample candidate hashes. External research confirms BLOCKHASH is a recognized "weak source of randomness from chain attributes," and that Base has run Pectra/Isthmus (hence EIP-2935) since May 2025. This is inherent to any future-blockhash design on Base and only affects peer/reference *pairings* (an information-score effect), never votes, the verdict, or a payout forge.
  2. **Stale-seed re-roll ratchet (Low):** the EIP-2935 window is 8191 blocks — on Base's ~2-second blocks that is only ~4.5 hours, not the ~27 hours quoted for L1's 12-second blocks. A coalition that dislikes the known draw and can keep a snapshot live-but-not-Finalized (e.g. self-challenged, which also blocks the 1-hour timeout) past ~4.5 hours forces a re-arm and a fresh unpredictable draw — a mild reject-known/accept-unknown positive-selection bias, strictly weaker than v3.
- **Fix (optional):** derive re-armed blocks deterministically (`seedBlock + k·8191`) so staleness cannot be *timed*; and document both residuals as named trust assumptions alongside the oracle model. The clean removal of both would be a drand/VRF-sourced seed, already on the roadmap discussion.

### 6P-7 (Informational). Open-eligibility bounty recapture: documented and accepted, with two economics footnotes worth stating exactly

- `a493feecc` removed the 500-USDC verified-human floor (the fourth-pass remediation for incentives M-1) so bounties are open-eligibility at any size. The subagent verified this is a *documented* accepted decision — README, whitepaper, SDK/AI docs, `docs/design-review-2026-07.md` M-1, and `docs/incentives-remediation-plan-2026-07.md` were all updated consistently; no surviving doc claims the floor exists. So this is an accepted residual, **not** a stated-policy/code contradiction. Two footnotes keep the documented economics exact: (1) an asker can register their own bonded frontend and route alts through it to recover even the frontend fee — and after `cc28c293b` those earned fees exit slashability in 1 hour — so the residual recapture cost is essentially just gas; (2) the only size-scaled control left is the 5/8-voter participant floor, which raises the *alt count*, not the marginal cost per dollar (recapture yield per alt rises with bounty size). Cluster-oracle pinning is effectively mandatory at launch but, by its own stated limits, does not discount independent-looking fresh wallets. Recommend one explicit sentence in the README / incentives plan that large open bounties are recapture-exposed by design.

### 6P-8 (Informational). Smaller hygiene items

- **Sibling skipped-flag hygiene (SC-6P-6 residual):** `QuestionRewardPoolEscrowQualificationLib.sol:371-373` zeroes `pendingPreQualificationRejectedRounds` on pool completion but leaves younger rounds' `preQualificationRejectedRound` flags set with no clearing path. Harmless at `requiredSettledRounds == 1`; a latent stale-bypass hazard if multi-round pools return.
- **`skipPreQualificationSnapshotlessClusterRound` is now permissionless** (`db3dd34c8` dropped the admin role). Verified benign — grief-skipping a source-ready round before the first proposal still leaves it fully qualifiable via the FIFO bypass — but worth a regression test asserting a grief-skipped round still pays.
- **Event forensics:** `RbtsSeedCaptured` emits the raw block number in a field named `entropy` and re-emits on re-arm (indexers can't distinguish arm/re-arm); the rejected-snapshot rating skip reuses `RATING_SNAPSHOT_NO_PROPOSAL_SKIP_REASON` (can't distinguish "never proposed" from "rejected"); `RatingSnapshotApplied` library-vs-registry ABI drift from the fifth pass (5P-6) — re-check it was picked up. Cosmetic but cheap.
- **`CLAIM_GRACE` still duplicated** between `QuestionRewardPoolEscrowPoolActionsLib` and `QuestionRewardPoolEscrow.sol` (fifth-pass 5P-8) — two constants that must move together.
- **Skipped-round late rating evidence is front-runnable** (`ContentRegistryRatingSnapshotLib.sol:191-198, 224`): a party fearing a skipped round's down-evidence can apply the next round's snapshot first and permanently block the late one. Ordering-over-completeness is the stated design; document that late recovery is best-effort/first-come.

## External research (Claude Fable 5, web)

- **OP-Stack / Base BLOCKHASH randomness (seed v4):** BLOCKHASH is a recognized weak randomness source from chain attributes, and on OP-Stack chains the L2 block hash is sequencer-constructed, so a colluding Base sequencer can grind the v4 seed. This is the correct named residual for 6P-6. Sources: [EIP-4399 (PREVRANDAO rationale / BLOCKHASH weakness)](https://eips.ethereum.org/EIPS/eip-4399), [OP Stack derivation spec](https://specs.optimism.io/protocol/derivation.html).
- **EIP-2935 on Base (seed staleness window):** Base activated the Isthmus hardfork bringing Pectra features (including EIP-2935 historical block hashes) in May 2025, so the fallback path the seed relies on is live. The 8191-block history window is ~27 hours at L1's 12-second blocks but only **~4.5 hours on Base's ~2-second blocks** — the correct figure for the 6P-6 re-roll-ratchet timing. Sources: [EIP-2935](https://eips.ethereum.org/EIPS/eip-2935), [Optimism brings Pectra to the Superchain](https://www.optimism.io/blog/optimism-brings-ethereum-s-pectra-upgrade-to-the-superchain), [Optimism Pectra changes notice](https://docs.optimism.io/notices/pectra-changes).
- Peer-prediction (RBTS/COCM) and drand/tlock production-status research from prior passes is unchanged and not repeated (`docs/design-review-2026-07.md`, `docs/design-review-fifth-pass-2026-07-02.md`).

## Notable non-findings (checked, no issue)

Proposal-scoped windows are snapshotted per proposal and read consistently on every reject/consume path (no retroactive governance effect in either direction). Deadline boundary semantics are exact-complementary across challenge, epoch veto, child effective veto, RBTS timeout, and FeedbackBonus award/reclaim — no same-second dual-state anywhere. `setOracleTimingConfig`/`setOracleConfig` are CONFIG_ROLE-gated, reject zero windows, bound at 3d/7d, and emit events. Counter symmetry across the whole skipped/recovered/reopened/abandoned machinery holds (all increments flag-guarded, decrements flag-guarded, duplicates revert). Every terminal state has a fund-conserving exit; complete-vs-unallocated refund branches conserve funds on every path traced. FeedbackBonusEscrow 1-hour timing landed on the contract side and matches the ponder mirror. Seed v4 EIP-2935 integration uses the correct deployed address and 8191 window with a revert-tolerant staticcall; the scoring set is frozen strictly before the seed block exists; retries are idempotent. Rating provisional-skip apply is strictly monotonic (no double-apply, no out-of-order). LaunchDistributionPool anchor/credential logic is independent of `bountyEligibility` — the eligibility removal breaks nothing there. The readiness assertion (`2·challenge + veto + ops ≤ 1h`) is enforced off-chain in `scripts/readiness-core.mjs`, as the plan specifies (nothing on-chain stops governance later configuring 3d/7d — an accepted control-plane choice).
