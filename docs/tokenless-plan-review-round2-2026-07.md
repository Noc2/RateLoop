# Multi-Agent Review of the Tokenless Plan — Round 2 (July 2026)

**Status:** 2026-07-12. Second five-lens review, after the v3/UX/DAC7/tier revisions: rater UX, funder & agent UX/DX, post-v3 consistency, adversarial weakest-link (operations under failure), and spec-readiness (week-1 implementability). **Disposition** codes as in [round 1](tokenless-plan-review-multi-agent-2026-07.md): *Fixed* (text corrected this pass), *Amended* (design/plan changed this pass), *Gate* (added to Phase −1/0), *Open* (accepted, tracked), *Rejected*.

## The eight most important findings

1. **[Process — HIGH] No document ever considered a pre-protocol pilot; Phase 0 sequencing is inverted.** ~6–12 months of legal gates now precede code for a product with zero validated demand, while a **managed pilot under an ordinary services agreement** (20–50 invited freelancer raters, sealed rounds via a plain web app, SEPA/Stripe payouts, 2–3 design partners on the confidential-pretesting wedge) sidesteps nearly every legal gate and *empirically produces* the numbers Phase 0 tries to estimate (wages, time-to-quorum, quorum-failure rates, gold hit-rates, willingness-to-pay, the quality benchmark). **Rejected (founder decision, 2026-07-12):** no centralized SEPA pilot — the full system is implemented immediately with AI and tested live; legal uncertainties tracked in parallel, change-or-abandon risk accepted. The plan's execution-philosophy section records this; Phase 0 reframed as a non-blocking parallel backlog with a short pre-real-money list.
2. **[Spec — BLOCKER] Sub Accounts vs per-round one-time keys is a real contradiction.** TL;DR 7's "Sub Accounts + Auto Spend Permissions for popup-free voting" was written before the fund-free redesign: voting moves no funds (Spend Permissions are irrelevant) and a Sub Account is persistent and linkable (voids §A.6's unlinkability). **Amended:** authorization decoupled from submission — the contract verifies an ecrecover signature from the voucher-bound vote key over the commit payload and ignores `msg.sender`; commits are relayed (Hawig's sponsored relayer, any relayer, or self-funded) with no per-key account deployment; Sub Accounts/Spend Permissions remain for **funder** flows only.
3. **[UX — P0] The rater surface is undecided and the wallet stack conflicts with the likeliest acquisition channel.** World App Mini Apps (MiniKit) allow no external wallets; Base Account needs a popup that in-app webviews handle poorly — the plan silently presumes a mobile-browser PWA while the identity story presumes World App users. **Resolved (founder decision):** the World App channel is deprioritized entirely (product targets beyond existing World ID users); surface = mobile-web PWA on **Base** (World Chain evaluated and declined — plan Part B), World App only as an optional verification handoff. Finding 2's relayed-commit design helps here: vote submission no longer requires wallet popups at all; the wallet only matters at payout.
4. **[Opsec — HIGH] The docs analyze every system in its healthy state.** The vote-key↔rater mapping DB is a **standing deanonymization key for an immutable public record** (breach = irreversible Art. 33/34 event linking every opinion to names via the DAC7 vault) — stated nowhere; the solo founder is single admin of issuer key, DAC7 filing deadlines, sanctions desk, and takedown clocks with no continuity story; the issuer key needs KMS custody + an issuance-rate circuit breaker. **Gate:** a Phase 0 "operational resilience & incident response" deliverable (breach model as the DPIA headline scenario, per-rater-key encryption of the mapping so Hawig holds ciphertext, envelope-encrypted DAC7 vault on a deletion schedule, backup/DR, sealed continuity plan, degradation modes in TRUST.md).
5. **[Lifecycle — HIGH] Illegal content in a live sealed round has only an accidental kill path, and rater age is never established.** **Amended:** pre-round moderation screen (content is served from Hawig storage anyway); "round kill = de-pin + de-list + funder-eligibility revocation + quorum-miss refund" documented and invariant-tested, with fee retention on ToS-violating content (don't refund the offender's fee); 18+ added to the eligibility matrix; JuSchG/JMStV added to the counsel list.
6. **[Mechanism/UX — HIGH] Platform failures were billed to raters.** "An unrevealable ciphertext simply earns nothing" assigns keeper/infra fault to the rater; drand is a single point of failure with no fallback; a Base sequencer outage mid-window breaks round fairness. **Amended:** self-reveal fallback (raters' clients keep plaintext+nonce; reveal-grace accepts self-reveals if the beacon is late), "beacon dead ⇒ full refund" invariant, off-contract make-good policy (treasury credit of expected base share) for keeper-fault reveal failures, sequencer-health runbook (pause new round creation when degraded), and honest TRUST.md lines: sealing trust rests on the drand threshold; funds are USDC and inherit **Circle's freeze/blacklist power** — the honest claim is "no *operator* path to funds," not "nobody."
7. **[Demand-side UX — HIGH] The plan has a rater-UX invariant but no funder/agent equivalent, and the Phase 0 economics table is never connected to the ask-creation UI.** **Amended:** Phase 1 gains a funder golden-path deliverable — presets derived from the economics table ("Quick pulse — 15 live humans, ~30 min, $25") instead of four raw knobs, preview-as-rater (doubles as the § 312j info block), pilot-round primitive, prepaid **workspace balance via a regulated partner as the primary B2B rail** (partner decision added to Phase 0), quorum-refund advertised as "filled or fully refunded," compliant **"Panel Verdict" share artifact** (the transparency log as funder-facing proof), tiered analytics SLOs (fast automated pass for low tiers; deep correlation reserved for high-assurance), and the GPT-as-judge benchmark as the demand-side sales asset. API DX additions: webhooks + polling + MCP long-wait pattern, idempotency keys on `ask`, free unauthenticated quote endpoint, permanent sandbox with simulated panels, versioned JSON-Schema result payloads with a verdict-status enum, machine-readable SLOs, and an API-key + prepaid-balance mode so agents can integrate without a wallet.
8. **[Spec — BLOCKERS] Four contract-level decisions had no schema; defaults are now recorded** (Part B): voucher struct `{voteKey, contentId, roundId, nullifier, tierId, expiresAt}` EIP-712-signed, engine checks signature + expiry + nullifier-unused + voteKey signature over the commit; the engine immutably trusts the **issuer contract**, which keeps a narrowly-scoped signer-rotation admin — explicitly documented as the one retained role, whitelisted in the assert-no-admin script (the earlier "rotation = redeploy the issuer" contradicted the immutable engine reference); nullifier base = provider-stable per-identity ID through a never-rotated derivation secret held separately from the signing key; engine→escrow interface = paginated `settleRound(…, cursor, count)` storing per-commit `weightBps` + escrow-side `claimWeight()` view + `heldFeeAmount` pushed at settlement/refunded on quorum-miss; payout commitment = `hash(payoutAddress, salt)` with `claim(commitKey, payoutAddress, salt)` **callable by anyone** (auto-claim without custody; self-claim fallback); **unclaimed shares revert to the funder after the stale grace**; per-identity caps enforced at voucher issuance and published (they cannot bind on-chain — stated honestly); advisory tier uses the same commit shape minus voucher/nullifier/payout-commitment; registry read at commit is voucher-only (registry shrinks to the confidential-lane gate); the keeper "skip reveals for revoked identities" idea is deleted (bypassable theater); testnet enablement (keeper 84532 support, Sepolia USDC, testnet paymaster/facilitator) added as a task; a Foundry gas benchmark for commit/settle becomes a Phase 1 acceptance criterion.

## Net assessment

Round 2's theme is the inverse of round 1's: the *decisions* now hold up (no reviewer challenged tokenless/keyless/stake-free/on-chain voting/gates-not-payment-complexity), but the plan was still a regulatory-statics document. What round 2 added: operations under failure (breach, continuity, beacon/sequencer, freeze, kill path), the demand side as a designed product rather than an API bullet point, four contract-level schemas that would have forced week-1 design meetings, one real architecture contradiction (Sub Accounts vs one-time keys) resolved by decoupling authorization from submission — and, most consequentially, **Phase −1**: the managed pilot that can test the wedge and produce the economics data while most of the legal machine stays parked. The recommended reading order for anyone joining now: Phase −1 decision → round-1 doc + addendum → this doc → the plan.
