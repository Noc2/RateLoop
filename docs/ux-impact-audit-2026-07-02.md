# UX Impact Audit - Changes Since 2026-07-01

Scope: commits from 2026-07-01 00:00 through 2026-07-02, plus the current uncommitted working-tree changes observed during this audit.

Short answer: yes. The most material UX change is that normal economic finality for rewards now depends on oracle-backed correlation payout snapshots and, on the current path, waits for the payout-root finalization veto window. There are also smaller cases where users or operators see extra confirmation, credential, or recovery steps.

Status note, 2026-07-02: this audit captured the pre-remediation UX impact. The fresh-deployment launch posture now uses 15-minute oracle challenge windows, a 15-minute finalization veto window, keeper/readiness budget checks for a 1-hour healthy-path finality target, 1-hour Feedback Bonus award-decision timing, and 1-hour frontend fee withdrawals while preserving the 14-day frontend stake unbonding period. Challenged, rejected, missing-proposer, or recovery paths remain exceptional states and are not part of the 1-hour normal UX promise.

## Confirmed UX-impacting changes

| Area | Change | Who feels it | UX effect | Severity |
| --- | --- | --- | --- | --- |
| LREP stake rewards and public result finality | `RoundVotingEngine.settleRound` now closes the public verdict into `SettlementPending` and waits for `applyRbtsSettlementSnapshot` before the round becomes `Settled`. `RoundRbtsSettlementSnapshotLib.applySnapshotWeights` rejects finalized RBTS snapshots until `finalizedAt + FINALIZATION_VETO_WINDOW`; `ClusterPayoutOracle.FINALIZATION_VETO_WINDOW` is 7 days. | Raters, askers, agents reading result packages | Getting paid from staked LREP now waits for the RBTS settlement root, its challenge/finalization, the 7-day veto window, keeper/application txs, and any cleanup. The result package also remains `pending` with `wait_for_settlement`, so agents do not get a final answer as soon as the public verdict closes. | High |
| Cluster-backed question and bundle bounty claims | `QuestionRewardPoolEscrowQualificationLib` and bundle qualification require finalized payout snapshots outside the same veto window before a cluster-backed round/set qualifies. The claim call verifies a finalized root, but normal claimability depends on the qualification row existing first. | Bounty-eligible raters and askers funding cluster-backed pools | Bounties that use the cluster payout oracle are not claimable immediately after the snapshot finalizes; they wait through the veto window before qualification can make them appear in claimable rewards. | High |
| Product copy timing drift | `protocolFacts.usdcBountyPayoutTimingTooltip` still says the happy-path payout wait is roughly 2-4 hours, but the current contract/keeper path waits through the 7-day veto window for RBTS/public-rating application and cluster-backed qualification. | Raters, askers, docs readers | Users may be told to expect hours while the implementation can require more than a week before rewards/results are ready. | High |
| Bounty eligibility choice | The 500,000,000 atomic-unit Proof-of-Human requirement has been removed. Direct, escrow, bundle, Next submit UI, agent metadata, lint, and x402 payload paths now preserve the creator's selected bounty eligibility, including `0` for everyone. | Askers and raters | Askers can choose "Everyone" or Proof of Human for bounty claims at any bounty size. Everyone can answer by default; unverified raters only need credentialing when the asker explicitly chooses a credential-gated bounty. | None |
| Private context mode switches | Manual submit and agent handoff draft editing now show a confirmation dialog when switching to private context would remove public context URL, YouTube URL, or uploaded public images. They also show a restore notice afterward. | Askers and agents/operators editing asks | This adds one extra click only when the mode switch would drop public context. It protects against silent data loss, but it is still an added interaction. | Low |
| Agent ask validation | Agent/x402 ask paths now reject more cases before handoff/submission: image plus video in the same question, bundles above the supported question count, and head-to-head templates used with multiple questions. | Agents creating asks | Some previously accepted drafts now require edits before a handoff can be created or submitted. This is mostly early failure, not a post-payment wait. | Low |
| Recovery and refund edge cases | Current recovery edits block ordinary `refundExpiredRewardPool`/`refundInactiveRewardPool` while recovered or prequalification-rejected rounds are pending. Callers must use recovery-specific functions with round IDs, and snapshotless cluster skips are admin-only. | Funders/operators in rejected/missing snapshot scenarios | Edge-case refunds can require an additional governance/operator action and a more specific refund call before funds return. Normal happy-path raters are not affected. | Medium |
| Frontend operator settings | Frontend registration now exposes access recorder setup/clear controls and shows ETH balances for operational wallets. | Frontend operators | Adds admin controls and an operational wallet setup surface. It is not normal rater/asker friction, but it is additional operator work for confidentiality logging readiness. | Low |

## Not counted as UX regressions

- Several July 1 UI changes reduce friction or clarify state: expired handoff recovery guidance, duplicate handoff recovery, hiding exhausted free-transaction allowance, mobile wallet balance stacking, removing image-signature status rows, and clearer answer-window/reward deadline chips.
- The private-context confirmation is intentionally scoped; it only appears when the user has public context that would be removed.
- The removed high-value Proof-of-Human rule no longer adds credentialing work unless the asker explicitly chooses a credential-gated bounty.

## Recommended follow-ups

1. Decide whether the 7-day payout-root veto wait is acceptable for launch UX. If yes, update public docs/tooltips/result-package language to say rewards can wait at least the veto window. If no, revisit the contract flow before deployment.
2. Add explicit UI copy/ETA for `Settlement pending`, including the root status, veto-window end, and whether cleanup is still blocking settlement.
3. Add monitoring for rounds or reward pools stuck in settlement/qualification longer than challenge window plus veto window plus keeper polling.
4. Keep bounty eligibility copy explicit: everyone can answer by default, and credentialing affects payout claims only when the asker selects a credential-gated bounty.
5. For recovery paths, consider operator runbook docs that map each stuck state to the exact recovery/refund function and required round IDs.
